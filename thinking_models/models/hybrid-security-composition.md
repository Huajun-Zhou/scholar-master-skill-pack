# Thinking Model: 混合安全原语组合

## 1. 现象/缺口

单一密码学方案虽然有完整的数学安全证明，但在实际系统中往往存在明显的性能瓶颈。纯FHE的自举消耗>90%时间，纯MPC的乘法在线通信开销极大，纯TEE受限于enclave内存。

## 2. 问题重构

从"选择一种密码方案保护全部计算"重构为"分析完整计算图的运算类型分布，为每种运算选择最优原语，设计低开销切换协议连接不同原语域"。

## 3. 关键假设

- 计算图的结构在编译/部署前已知
- 不同安全原语的信任假设可以接受（混合信任模型）
- 原语切换的边界不会引入新的可利用漏洞

## 4. 方法机制

1. **计算剖分**：将完整计算拆解为运算类型图
2. **瓶颈识别**：定位单一方案下的性能瓶颈点（FHE自举/MPC乘法/TEE内存）
3. **互补选择**：线性→ASS(零通信)；非线性→FSS；深层次→TEE
4. **切换设计**：在原语边界设计低开销安全域转换协议
5. **混合证明**：在组合安全模型下证明整体安全性

## 5. 实验验证

- 原语级别的性能基准（cpu cycles / bytes）
- 混合方案 vs 单一方案的整体性能对比
- 安全域切换开销占总开销的比例

## 6. 不适用场景

- 纯同类型运算的计算任务（无混合收益）
- 安全合规要求单一方案
- 切换协议本身复杂度超过收益

## 7. 迁移模板

问自己：
1. 我的计算图中哪些是线性运算？哪些是非线性？
2. 单一方案的最大瓶颈在哪？（自举/通信/内存）
3. 哪种原语在这个瓶颈点上最有效？
4. 两种原语的信任假设是否兼容？
5. 切换协议引入的开销是否显著低于消除的瓶颈收益？
